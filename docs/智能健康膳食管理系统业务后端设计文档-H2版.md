# 智能健康膳食管理系统业务后端设计文档（H2 版）

## 1. 文档说明

本文档面向课程大作业开发与答辩，描述“智能健康膳食管理系统”的业务后端设计。系统采用 Spring Boot + H2 嵌入式内存数据库实现，重点完成用户健康档案、健康目标周期、食材菜谱、膳食计划、计划执行反馈、膳食记录、营养风险分析、营养统计、AI 解析菜谱结果管理和社区分享等业务能力。

与复杂生产系统不同，H2 版后端强调“可运行、可演示、逻辑完整、开发成本可控”。数据库默认使用 H2 内存模式，启动时通过初始化脚本加载演示数据，适合课程展示、答辩和本地开发。

AI 图片识别、ResNet18 + Food-101 模型推理和 LLM 结构化菜谱生成由独立 `ai-service` 提供。业务后端只负责调用 `ai-service`、校验返回 JSON、保存 AI 菜谱结果，并把结果展示给前端。

## 2. 建设目标

- 支持用户注册、登录、JWT 鉴权和个人资料管理。
- 支持健康档案、饮食偏好、过敏源、慢性病、饮食禁忌管理。
- 支持食材库和菜谱库管理，并根据食材用量自动计算菜谱营养。
- 支持基于健康目标、热量目标、过敏源和偏好的膳食计划生成。
- 支持阶段性健康目标周期管理，包括目标拆解、体重趋势、完成度和偏离预警。
- 支持菜谱适配评分，根据用户健康目标、过敏源、偏好和营养结构评估菜谱匹配度。
- 支持膳食计划执行反馈，记录计划完成、跳过、替换原因，并形成执行率统计。
- 支持替换推荐机制，在计划菜谱不合适时推荐同餐次、相近热量且更符合偏好的替代菜谱。
- 支持营养风险规则引擎，基于控糖、高血压、减脂、增肌等场景生成风险提示。
- 支持每日膳食记录、营养汇总、热量缺口、宏量营养素占比分析。
- 支持营养周报/月报，汇总达标天数、趋势变化、风险次数和个性化建议。
- 支持调用独立 `ai-service`，将图片识别和 LLM 结果保存为 AI 解析菜谱。
- 支持 AI 解析菜谱转膳食记录、转正式菜谱、分享到社区。
- 支持社区帖子、评论、点赞、收藏和管理员审核。
- 使用 H2 内存数据库降低部署成本，系统启动即可演示完整业务流程。

## 3. 非目标说明

- 不实现真实医疗诊断，所有健康建议仅作为饮食参考。
- 不实现真实线上支付、物流、短信验证码等外部商业能力。
- 不在业务后端训练 AI 模型，模型训练和推理由独立 `ai-service` 负责。
- 不以生产环境高可用为目标，课程项目阶段优先保证功能闭环和逻辑自洽。
- H2 内存模式下数据随服务重启重置；如需保留数据，可切换为 H2 文件模式。

## 6. 后端分层设计

```text
controller
  接收 HTTP 请求，完成参数校验，返回统一响应

service
  实现业务规则、事务控制、权限判断和跨模块编排

repository
  使用 Spring Data JPA 访问 H2 数据库

domain / entity
  定义数据库实体和核心领域对象

dto
  定义请求、响应和分页对象，避免实体直接暴露给前端

client
  封装 ai-service 调用、文件访问等外部能力

security
  JWT 生成校验、用户认证、接口权限控制

common
  统一响应、异常处理、枚举、工具类
```

## 8. 角色与权限设计

| 角色     | 标识      | 权限范围                                                                                      |
| -------- | --------- | --------------------------------------------------------------------------------------------- |
| 游客     | ANONYMOUS | 注册、登录、查看公开菜谱、查看公开帖子                                                        |
| 普通用户 | USER      | 管理个人健康数据、健康目标周期、膳食计划、执行反馈、膳食记录、AI 解析菜谱、营养报告、社区互动 |
| 管理员   | ADMIN     | 管理用户、食材、菜谱、帖子、评论、系统演示数据                                                |

权限原则：

- 用户只能访问和修改自己的健康档案、偏好、膳食记录、AI 菜谱和体重记录。
- 公开菜谱和公开帖子允许游客查看。
- 管理员可以查看全站内容，但用户邮箱、手机号等敏感字段需要脱敏。
- AI 解析菜谱结果属于用户私有数据，只有作者本人和管理员可查看。

## 9. H2 数据库设计

### 9.1 H2 配置建议

内存模式：

```yaml
spring:
  datasource:
    url: jdbc:h2:mem:indras;MODE=MySQL;DATABASE_TO_UPPER=false;DB_CLOSE_DELAY=-1
    driver-class-name: org.h2.Driver
    username: sa
    password:
  h2:
    console:
      enabled: true
      path: /h2-console
  jpa:
    hibernate:
      ddl-auto: create-drop
    show-sql: true
```

文件模式：

```yaml
spring:
  datasource:
    url: jdbc:h2:file:./data/indras;MODE=MySQL;DATABASE_TO_UPPER=false
  jpa:
    hibernate:
      ddl-auto: update
```

课程演示推荐使用内存模式，配合 `data.sql` 初始化用户、食材、菜谱、帖子等演示数据。

## 10. 核心业务流程

### 10.1 注册登录流程

```text
前端提交注册信息
  -> 后端校验用户名和邮箱是否重复
  -> BCrypt 加密密码
  -> 创建 sys_user、user_profile、默认 user_preference
  -> 用户登录
  -> 生成 JWT
  -> 前端携带 Token 访问业务接口
```

### 10.2 健康档案维护流程

```text
用户填写身高、体重、生日、活动水平、健康目标
  -> 后端计算年龄、BMI、基础代谢率 BMR
  -> 根据活动水平计算 TDEE
  -> 根据健康目标生成每日热量目标
  -> 保存 health_profile
  -> 后续膳食计划和 AI 解析菜谱使用该上下文
```

### 10.2.1 健康目标周期管理流程

```text
用户创建减脂、增肌、控糖或维持目标周期
  -> 后端校验开始日期、结束日期和目标体重是否合理
  -> 根据当前体重、目标体重和周期长度计算每周目标变化
  -> 结合活动水平生成周期内每日热量目标
  -> 保存 health_goal_cycle，并将其作为 ACTIVE 周期
  -> 用户录入体重记录后重新计算目标完成度
  -> 如果连续多次偏离目标趋势，生成偏离预警和调整建议
```

### 10.3 菜谱营养计算流程

```text
用户或管理员创建菜谱
  -> 添加食材和用量
  -> 后端查询每种食材每 100g 营养
  -> 按用量折算热量、蛋白质、脂肪、碳水
  -> 汇总写回 recipe
  -> 菜谱进入 DRAFT 或 PENDING 状态
```

### 10.4 膳食计划生成流程

```text
用户选择日期和目标
  -> 查询健康档案、当前健康目标周期、过敏源、禁忌、偏好
  -> 筛选 ONLINE 菜谱
  -> 排除含过敏食材的菜谱
  -> 根据健康目标过滤高糖、高脂、高盐菜谱
  -> 对候选菜谱计算用户个性化适配评分
  -> 按早餐、午餐、晚餐、加餐分配热量比例
  -> 选择热量接近目标、风险较低且偏好匹配的菜谱组合
  -> 生成 meal_plan 和 meal_plan_item
  -> 执行营养风险规则检查
  -> 返回计划评分和推荐原因
```

### 10.4.1 膳食计划执行反馈流程

```text
用户查看今日膳食计划
  -> 对每个餐次标记已完成、部分完成、跳过或已替换
  -> 后端保存 meal_plan_feedback
  -> 已完成或部分完成的餐次可自动生成 meal_record
  -> 重新计算当日营养摄入和计划完成率
  -> 将执行率写入营养周报/月报统计口径
```

### 10.4.2 替换推荐流程

```Java
用户对计划中的某个餐次发起替换
  -> 后端读取原计划餐次、原菜谱营养和用户替换原因
  -> 筛选同餐次 ONLINE 菜谱
  -> 排除过敏源和饮食禁忌冲突菜谱
  -> 按热量差、适配评分、偏好匹配和最近重复度排序
  -> 返回 Top-N 替换候选及推荐原因
  -> 用户确认替换后更新 meal_plan_item
  -> 保存 meal_plan_replace_log，并重新计算计划总热量和计划评分
```

### 10.4.3 营养风险规则执行流程

```text
用户生成计划、记录膳食或查看营养汇总
  -> 后端读取用户健康档案和健康目标周期
  -> 匹配适用的 nutrition_risk_rule
  -> 对热量、蛋白质、脂肪、碳水、钠等指标执行规则判断
  -> 生成 nutrition_risk_result
  -> 前端展示风险等级、原因和调整建议
```

### 10.5 AI 解析菜谱流程

```text
用户上传食物图片或输入想吃的内容
  -> Spring Boot 保存图片或接收用户文本
  -> 查询用户健康档案、偏好、过敏源、饮食禁忌
  -> 调用 ai-service /ai/v1/recipes/parse 或 /parse-from-image
  -> ai-service 返回结构化菜谱 JSON
  -> Spring Boot 校验 JSON 字段和数据类型
  -> 检查 ingredients 是否包含用户过敏源
  -> 保存 ai_recipe 和 ai_recipe_step
  -> 前端展示菜谱名称、营养估算、适配原因、做菜步骤
```

### 10.6 AI 菜谱转膳食记录流程

```text
用户确认 AI 解析菜谱
  -> 选择日期、餐次、份量比例
  -> 后端读取 ai_recipe 的 nutrition_json
  -> 按份量比例折算营养
  -> 创建 meal_record
  -> 来源标记为 AI_RECIPE
  -> 更新当天营养统计
```

### 10.7 社区分享流程

```text
用户上传社区图片
  -> 后端校验格式、大小和登录身份
  -> 图片保存到 uploads/community/
  -> 新增 community_post_image(TEMP)
  -> 返回 imageId、imageUrl

用户手工发帖
  -> 提交 title/content/tags/imageIds
  -> 后端校验 imageIds 数量 1-9，且均为本人 TEMP 图片
  -> 创建 community_post
  -> 绑定 community_post_image.post_id，状态改为 ACTIVE，按 imageIds 顺序写入 sort_no
  -> 帖子进入 PENDING 审核状态

用户从 AI 菜谱发帖
  -> 后端读取 ai_recipe.source_image_url
  -> 若 useAiSourceImage=true，则将 AI 原图作为首图
  -> 合并用户补充上传的 imageIds，总数不得超过 9
  -> 创建 community_post 和 community_post_image 记录
  -> 帖子进入 PENDING 审核状态

游客查看社区
  -> 列表只返回 coverImageUrl 和 imageCount
  -> 详情返回完整 images[] 相册
  -> 历史无图帖子允许兼容读取
```

### 10.8 临时图片清理策略

- 社区图片上传后先落为 `TEMP` 状态。
- 发帖成功后切换为 `ACTIVE`，并绑定 `post_id`。
- 超过 24 小时未绑定帖子的 `TEMP` 图片可由定时任务清理。
- H2 演示版统一存放在 `uploads/community/`，图片公开 URL 由 Spring Boot 静态资源映射提供。

### 10.9 营养周报/月报生成流程

```text
用户选择周报或月报统计周期
  -> 后端汇总周期内 meal_record、meal_plan_feedback 和 nutrition_risk_result
  -> 计算平均热量、三大营养素均值、记录天数、达标天数和执行完成率
  -> 统计高频风险、最常吃菜谱和目标趋势
  -> 生成 nutrition_report
  -> 前端通过折线图、环形图和建议列表展示报告
```

## 11. 业务规则设计

### 11.1 BMI 计算

```text
BMI = 体重 kg / (身高 m * 身高 m)
```

| BMI 范围        | 状态 |
| --------------- | ---- |
| `< 18.5`      | 偏瘦 |
| `18.5 - 23.9` | 正常 |
| `24.0 - 27.9` | 超重 |
| `>= 28.0`     | 肥胖 |

### 11.2 每日热量目标计算

基础代谢率可使用 Mifflin-St Jeor 公式：

```text
男性 BMR = 10 * 体重kg + 6.25 * 身高cm - 5 * 年龄 + 5
女性 BMR = 10 * 体重kg + 6.25 * 身高cm - 5 * 年龄 - 161
```

活动系数：

| 活动水平   | 系数  |
| ---------- | ----- |
| 久坐       | 1.2   |
| 轻度活动   | 1.375 |
| 中度活动   | 1.55  |
| 高强度活动 | 1.725 |

目标调整：

| 健康目标 | 调整                       |
| -------- | -------------------------- |
| 减脂     | TDEE - 300                 |
| 维持     | TDEE                       |
| 增肌     | TDEE + 300                 |
| 控糖     | TDEE - 100，并限制高糖食材 |

### 11.3 菜谱营养计算

```text
营养素总量 = Σ(食材每 100g 营养素值 * 食材用量 g / 100)
```

计算结果写回菜谱表，便于搜索、推荐和营养统计。

### 11.4 膳食计划评分

```text
计划评分 =
  热量匹配分 * 0.35
  + 营养均衡分 * 0.30
  + 偏好匹配分 * 0.20
  + 多样性分 * 0.15
```

评分规则：

- 热量越接近每日目标，热量匹配分越高。
- 蛋白质、脂肪、碳水比例越接近推荐范围，营养均衡分越高。
- 菜谱标签与用户偏好匹配越多，偏好匹配分越高。
- 最近 7 天重复出现的菜谱越少，多样性分越高。

### 11.5 菜谱适配评分

菜谱适配评分用于菜谱详情页、膳食计划生成和替换推荐。

```text
菜谱适配评分 =
  热量匹配分 * 0.30
  + 宏量营养素匹配分 * 0.25
  + 偏好匹配分 * 0.20
  + 目标匹配分 * 0.15
  + 风险安全分 * 0.10
```

评分规则：

- 热量越接近当前健康目标周期建议餐次热量，热量匹配分越高。
- 减脂用户优先高蛋白、低脂；增肌用户提高蛋白质权重；控糖用户降低高碳水菜谱得分。
- 命中用户喜欢的食材、烹饪方式或菜系时增加偏好匹配分。
- 命中过敏源直接判定为不可推荐；命中饮食禁忌时降低风险安全分。
- 适配分低于 60 时仅展示为普通菜谱，不进入自动计划和替换推荐候选。

### 11.6 替换推荐规则

替换推荐必须满足硬约束，再进行排序。

硬约束：

- 新菜谱餐次分类应与原餐次一致或兼容。
- 新菜谱不得包含用户过敏源。
- 新菜谱状态必须为 ONLINE。
- 新菜谱热量与原餐次目标热量偏差原则上不超过 30%。

排序因子：

- 与原菜谱热量差越小，排序越靠前。
- 菜谱适配评分越高，排序越靠前。
- 最近 7 天未吃过的菜谱优先。
- 如果用户替换原因为“不喜欢”，降低相同标签或相同主食材菜谱排序。
- 如果用户替换原因为“热量过高”，优先推荐低热量且蛋白质充足的菜谱。

### 11.7 膳食计划执行率计算

```text
计划执行率 = 已完成餐次数 / 计划餐次数 * 100%
```

计算规则：

- `COMPLETED` 按 1 次完成计算。
- `PARTIAL` 按 0.5 次完成计算。
- `SKIPPED` 按 0 次完成计算。
- `REPLACED` 如果替换后生成了膳食记录，按 1 次完成计算。
- 执行率用于健康目标周期复盘、营养周报/月报和后续推荐策略调整。

### 11.8 营养风险规则

营养风险规则采用“用户场景 + 营养指标 + 阈值”的方式配置，便于演示和扩展。

示例规则：

| 场景   | 指标   | 规则              | 风险等级 | 提示                         |
| ------ | ------ | ----------------- | -------- | ---------------------------- |
| 控糖   | 碳水   | 单餐碳水 > 90g    | WARNING  | 当前餐碳水偏高，建议减少主食 |
| 高血压 | 钠     | 单日钠 > 2000mg   | DANGER   | 钠摄入偏高，建议少盐         |
| 减脂   | 热量   | 单日热量 > 目标值 | WARNING  | 今日热量已超出减脂目标       |
| 增肌   | 蛋白质 | 单日蛋白质 < 目标 | INFO     | 蛋白质不足，建议补充优质蛋白 |

规则执行结果可用于菜谱适配评分、计划生成、营养报告和前端风险提示。

### 11.9 营养周报/月报统计规则

周报/月报以膳食记录为主，以计划执行反馈和风险结果作为补充分析。

- 达标天数：当天热量在目标值上下 10% 范围内，且无 DANGER 风险。
- 记录天数：周期内至少存在一条膳食记录的日期数。
- 平均营养：按记录天数计算热量、蛋白质、脂肪、碳水均值。
- 计划完成率：周期内所有计划反馈的加权完成率。
- 高频风险：按规则编码统计命中次数，取 Top 3 展示。
- 建议生成：根据目标周期、风险结果和趋势变化生成固定模板建议。

### 11.10 AI 菜谱校验规则

`ai-service` 返回结果后，业务后端必须二次校验：

- `recipeName` 不能为空。
- `ingredients` 必须是数组。
- `nutritionEstimate.calories` 必须大于 0。
- `cookingSteps` 必须是非空数组。
- 食材名称不得命中用户过敏源。
- 如果适配分低于 60，前端展示“谨慎选择”提示。
- LLM 原始返回保存在 `raw_response_json`，便于答辩演示和排错。
- 文本解析生成的 AI 菜谱 `source_image_url` 允许为空。
- 图片解析生成的 AI 菜谱必须保存 `source_image_url` 和 `source_image_key`，用于社区分享复用原图。
- AI 菜谱转社区帖子时，若没有原图，则必须额外上传社区图片。
- AI 原图与补充社区图片合并后总数不得超过 9 张。

## 13. 与 ai-service 的集成设计

### 13.1 调用边界

| 能力              | 业务后端 | ai-service |
| ----------------- | -------- | ---------- |
| 用户登录鉴权      | 是       | 否         |
| 健康档案查询      | 是       | 否         |
| 图片保存          | 是       | 可临时读取 |
| ResNet18 图片识别 | 否       | 是         |
| LLM 菜谱生成      | 否       | 是         |
| JSON 二次校验     | 是       | 是         |
| AI 菜谱落库       | 是       | 否         |
| 前端展示响应      | 是       | 否         |

### 13.2 调用流程

```text
AiRecipeController
  -> AiRecipeService
  -> HealthContextService 查询用户上下文
  -> AiServiceClient 调用 ai-service
  -> AiRecipeValidator 校验返回 JSON
  -> AiRecipeRepository 保存 ai_recipe
  -> 返回前端展示 DTO
```

### 13.3 ai-service 客户端接口

```text
POST /ai/v1/recipes/parse
POST /ai/v1/recipes/parse-from-image
GET  /ai/v1/health
```

调用超时建议：

| 类型         | 超时  |
| ------------ | ----- |
| 健康检查     | 2 秒  |
| 文本解析菜谱 | 20 秒 |
| 图片解析菜谱 | 30 秒 |

### 13.4 降级策略

- `ai-service` 不可用时，返回 `50200 外部 AI 服务不可用`。
- LLM JSON 不合法时，记录原始响应，返回 `50201 AI 返回格式错误`。
- 图片识别置信度过低时，仍允许用户手动输入菜名继续解析。
- AI 菜谱包含过敏食材时，后端拒绝保存为确认状态，并提示用户调整。

## 15. 安全设计

- 密码使用 BCrypt 加密保存。
- JWT 设置合理过期时间，前端通过 `Authorization: Bearer <token>` 调用接口。
- 用户资源访问必须校验 `user_id` 归属。
- 管理员接口统一使用 `/api/v1/admin` 前缀，并限制 ADMIN 角色。
- H2 Console 仅在本地开发环境开启，生产或答辩打包环境可关闭。
- 日志中不打印密码、JWT、ai-service 内部令牌和完整 Prompt。
- AI 解析菜谱只传递必要健康上下文，避免传输手机号、邮箱等无关个人信息。

## 21. 答辩展示建议

推荐演示主线：

1. 使用演示账号登录系统。
2. 查看或修改健康档案，展示 BMI 和每日热量目标自动计算。
3. 创建或查看当前健康目标周期，展示目标完成度和体重趋势。
4. 查看食材库和菜谱库，演示菜谱营养自动汇总和个人适配评分。
5. 生成今日膳食计划，展示推荐原因、计划评分和风险提示。
6. 对某个餐次触发替换推荐，展示候选菜谱排序和替换原因。
7. 提交计划执行反馈，展示完成率如何影响统计。
8. 上传食物图片或输入“想做减脂晚餐”，调用 AI 解析菜谱。
9. 展示 AI 返回的结构化菜谱：食材、营养估算、适配原因、做菜步骤数组。
10. 将 AI 菜谱转为晚餐记录。
11. 查看今日营养统计和营养风险提示。
12. 生成营养周报或月报，展示达标天数、趋势、风险次数和建议。
13. 将 AI 菜谱分享到社区。
14. 管理员审核帖子或菜谱，并维护营养风险规则。

答辩表述建议：

```text
本项目业务后端采用 Spring Boot + H2 嵌入式数据库，降低部署复杂度，
同时保留健康档案、健康目标周期、营养计算、菜谱适配评分、膳食计划、替换推荐、执行反馈、膳食记录、营养风险分析、AI 解析菜谱和社区分享等完整业务链路。
AI 能力不直接写死在后端中，而是通过独立 ai-service 接入；
后端负责查询用户健康上下文、调用 AI 服务、校验结构化 JSON、保存业务结果并驱动前端展示。
这样既保证了课程项目可演示性，也体现了模块解耦和业务逻辑复杂度。
```

## 22. 总结

H2 版业务后端以“简单部署、完整闭环、业务逻辑足够”为设计原则。系统使用 Spring Boot 承担核心业务编排，使用 H2 内存数据库降低环境依赖，通过初始化数据保证开箱即演示。业务复杂度集中体现在健康目标周期管理、菜谱营养折算、菜谱适配评分、膳食计划评分、替换推荐、计划执行反馈、营养风险规则引擎、营养周报/月报、AI 解析菜谱结果校验、AI 菜谱转膳食记录和社区分享等流程上。

AI 模型训练、Food-101 图片识别和 LLM 生成逻辑由独立 `ai-service` 负责，业务后端通过稳定接口集成该能力。整体设计既避免了单纯套壳 LLM 的薄弱感，又不会让课程项目陷入过重的数据库和部署复杂度。
