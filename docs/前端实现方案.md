# 智能健康膳食管理系统前端实现方案

## 1. 目标与原则

本文档基于 [前后端接口契约](./前后端接口契约.md) 设计 Vue3 前端的分模块实现方案。当前后端尚未实现，前端开发以 Mock 数据驱动，但必须保证数据访问层与 UI 解耦，后续联调时可以按接口逐步切换到真实 API。

核心原则：

- UI 组件只调用业务 Service，不直接感知 Mock.js、Axios 或真实后端地址。
- Mock 数据由 Mock.js 按接口契约随机生成，字段名、枚举值、分页结构、统一响应结构与契约保持一致。
- 每个接口至少覆盖正常响应、空数据、错误响应、超时四种场景。
- 数据源开关支持全局默认值和逐接口覆盖，联调时一条接口稳定后切一条，不影响其他页面开发。
- 所有接口统一走 `request` 封装，统一处理 `Authorization`、`code !== 0`、401 跳转、超时和错误提示。

## 2. 技术基线

当前前端为 Vite + Vue3 项目，建议在现有基础上补齐以下依赖：

```bash
npm install vue-router pinia axios mockjs element-plus @element-plus/icons-vue echarts dayjs
npm install -D @types/mockjs
```

建议技术分层：

| 层级         | 职责                                 | 建议目录                 |
| ------------ | ------------------------------------ | ------------------------ |
| 路由层       | 页面入口、权限元信息、懒加载         | `src/router`           |
| 状态层       | 登录态、用户信息、跨页面缓存         | `src/stores`           |
| 请求层       | axios 实例、响应解包、错误处理       | `src/services/request` |
| 业务 Service | 按领域暴露接口函数                   | `src/services/modules` |
| Mock Service | Mock.js 路由注册、场景控制、数据工厂 | `src/mock`             |
| 页面层       | 业务页面容器                         | `src/views`            |
| 组件层       | 可复用 UI 和领域组件                 | `src/components`       |
| 常量类型     | 枚举、接口路径、场景配置             | `src/constants`        |

## 4. 数据源与 Mock 开关设计

### 4.1 环境变量

在 `frontend/.env.development` 维护默认数据源：

```text
VITE_API_BASE_URL=http://localhost:8080/api/v1
VITE_USE_MOCK=true
VITE_MOCK_DELAY=300
VITE_MOCK_TIMEOUT_DELAY=8000
VITE_MOCK_DEFAULT_SCENARIO=success
```

`VITE_USE_MOCK=true` 表示默认启用 Mock.js 拦截。真实联调时不建议一次性关闭 Mock，而是通过逐接口配置切换。

### 4.2 逐接口数据源配置

新增 `src/api/scenario.js`，维护接口级策略：

```js
export const DATA_SOURCE = {
  MOCK: 'mock',
  REAL: 'real',
}

export const MOCK_SCENARIO = {
  SUCCESS: 'success',
  EMPTY: 'empty',
  ERROR: 'error',
  TIMEOUT: 'timeout',
}

export const endpointDataSource = {
  'GET /auth/me': DATA_SOURCE.MOCK,
  'POST /auth/login': DATA_SOURCE.MOCK,
  'GET /nutrition/today': DATA_SOURCE.MOCK,
}

export const endpointScenario = {
  'GET /recipes': MOCK_SCENARIO.SUCCESS,
  'GET /meal-plans/day': MOCK_SCENARIO.EMPTY,
  'POST /ai-recipes/parse': MOCK_SCENARIO.TIMEOUT,
}
```

切换原则：

- 开发期：`VITE_USE_MOCK=true`，大多数接口走 Mock。
- 单接口联调：只把目标接口设置为 `REAL`，其他接口继续 Mock。
- 联调稳定后：逐模块切换为 `REAL`，保留 Mock 场景用于页面状态回归。
- 生产构建：`VITE_USE_MOCK=false`，不注册 Mock.js。

### 4.3 请求适配

所有业务模块调用 `request`：

```js
// services/modules/recipe.service.js
import { request } from '../request/http'

export function fetchRecipes(params) {
  return request.get('/recipes', { params })
}
```

`request` 的职责：

- 自动拼接 `VITE_API_BASE_URL`。
- 自动注入 `Authorization: Bearer <token>`。
- 对统一响应解包，只把 `data` 返回给业务 Service 调用方。
- 当 `code !== 0` 时抛出统一业务异常。
- 当业务码为 `40100` 时清空登录态并跳转登录页。
- 对网络错误、超时错误和业务错误输出统一错误对象。

UI 调用示例：

```js
const pageData = await fetchRecipes({ page: 1, size: 10, keyword })
recipes.value = pageData.items
pagination.value = pageData
```

UI 只关心 `fetchRecipes` 的返回数据，不关心数据来自 Mock 还是真实后端。

## 5. Mock.js 架构

### 5.1 统一响应生成器

`src/mock/helpers.js`：

```js
import Mock from 'mockjs'

export function success(data) {
  return {
    code: 0,
    message: 'success',
    data,
    timestamp: new Date().toISOString(),
  }
}

export function failure(code = 40001, message = '参数校验失败', data = null) {
  return {
    code,
    message,
    data,
    timestamp: new Date().toISOString(),
  }
}

export function page(items, page = 1, size = 10, total = items.length) {
  return {
    items,
    page,
    size,
    total,
    pages: Math.ceil(total / size),
  }
}

export function pick(values) {
  return values[Mock.Random.integer(0, values.length - 1)]
}
```

### 5.2 场景控制

`src/mock/scenario-store.js`：

```js
import { MOCK_SCENARIO } from '../api/scenario'

export function resolveScenario(endpointKey) {
  return window.__MOCK_SCENARIO__?.[endpointKey] || MOCK_SCENARIO.SUCCESS
}

export function withScenario(endpointKey, handlers) {
  const scenario = resolveScenario(endpointKey)
  return handlers[scenario] ? handlers[scenario]() : handlers.success()
}
```

浏览器调试时可临时切换：

```js
window.__MOCK_SCENARIO__ = {
  'GET /recipes': 'empty',
  'POST /ai-recipes/parse': 'timeout',
}
```

### 5.3 四类响应场景

| 场景     | 用途                         | Mock 行为                                                | UI 验证点                        |
| -------- | ---------------------------- | -------------------------------------------------------- | -------------------------------- |
| 正常响应 | 主流程开发                   | `code: 0`，返回随机但格式正确的数据                    | 页面渲染、分页、图表、表单回填   |
| 空数据   | 首次使用或无搜索结果         | `code: 0`，列表 `items: []` 或详情 `null`          | 空状态、创建入口、无结果提示     |
| 错误响应 | 参数错误、权限不足、业务失败 | 返回`code !== 0`，如 `40001/40100/40300/42200/50200` | 错误提示、表单字段错误、登录失效 |
| 超时     | AI、报告、风险规则等耗时接口 | 延迟超过前端超时阈值或返回 Promise 长等待                | Loading、取消/重试、超时提示     |

### 5.4 Mock 数据工厂

每类实体独立工厂，保证字段契约集中维护：

```js
// mock/factories/recipe.js
import Mock from 'mockjs'
import { pick } from '../helpers'

export function createRecipe(overrides = {}) {
  return {
    id: Mock.Random.integer(1001, 9999),
    name: `${pick(['鸡胸肉', '番茄', '西兰花', '牛肉', '豆腐'])}${pick(['蔬菜碗', '炒蛋', '沙拉', '汤'])}`,
    description: pick(['高蛋白低脂餐', '家常快手菜', '适合控糖人群', '轻负担晚餐']),
    category: pick(['BREAKFAST', 'LUNCH', 'DINNER', 'SNACK']),
    difficulty: pick(['EASY', 'NORMAL', 'HARD']),
    cookMinutes: Mock.Random.integer(10, 60),
    servings: Mock.Random.integer(1, 4),
    totalCalorie: Mock.Random.integer(180, 650),
    totalProtein: Mock.Random.float(8, 50, 1, 1),
    totalFat: Mock.Random.float(2, 30, 1, 1),
    totalCarbohydrate: Mock.Random.float(10, 90, 1, 1),
    status: pick(['DRAFT', 'PENDING', 'ONLINE', 'OFFLINE']),
    favorite: Mock.Random.boolean(),
    suitabilityScore: Mock.Random.integer(60, 98),
    ...overrides,
  }
}
```

工厂要求：

- 枚举值只从 `constants/enums.js` 引用，避免散落硬编码。
- 日期统一使用 `utils/date.js` 生成 `YYYY-MM-DD` 或 ISO 字符串。
- 列表数据通过工厂批量生成，分页数据通过 `page()` 包装。
- 详情数据复用列表工厂并追加 `ingredients`、`steps` 等详情字段。

## 6. 模块拆分与实现顺序

### 第 0 阶段：前端基础设施

目标：让所有业务页面都能在统一骨架上开发。

实现内容：

- 安装依赖并接入 Vue Router、Pinia、Element Plus、ECharts。
- 建立 `request`、业务 Service、Mock Service、枚举常量和格式化工具。
- 建立主布局：侧边导航、顶部用户区、内容区域、移动端导航。
- 建立通用组件：分页表格、空状态、错误状态、加载状态、指标卡、图表容器、确认弹窗。
- 建立登录态 Store 和路由守卫。

验收：

- 任意 Service 可通过 Mock 返回统一响应。
- 可在浏览器控制台切换某接口为空数据、错误和超时。
- UI 页面没有直接导入 Mock 模块。

- 审核通过/拒绝。
- 日志为空。
- 重置演示数据失败和成功。

## 7. Service 设计规范

每个业务模块提供一组稳定函数名，页面只依赖函数签名。

示例：

```js
// services/modules/meal.service.js
export function generateDayMealPlan(payload) {}
export function fetchDayMealPlan(params) {}
export function fetchReplacementCandidates(planId, itemId, params) {}
export function replaceMealPlanItem(planId, itemId, payload) {}
export function submitPlanItemFeedback(planId, itemId, payload) {}
export function fetchDayMealRecords(params) {}
export function createManualMealRecord(payload) {}
```

规范：

- Service 函数使用业务命名，不暴露 HTTP 动词细节给页面。
- URL 路径集中从 `api/endpoints.js` 引用，避免散落字符串。
- 列表接口统一返回分页对象，不在 Service 内改字段名。
- 文件上传 Service 内部构造 `FormData`，页面只传 `File` 和可选 `prompt`。
- Service 不处理 UI 提示，只抛错；页面或全局错误处理器决定展示方式。

## 8. 页面状态规范

每个需要异步数据的页面至少实现以下状态：

| 状态    | 触发               | UI 表现                          |
| ------- | ------------------ | -------------------------------- |
| loading | 请求中             | 骨架屏、按钮 loading、图表加载态 |
| success | 正常有数据         | 正常渲染                         |
| empty   | 正常但无数据       | 空状态、创建或重试入口           |
| error   | 业务错误或网络错误 | 错误提示、重试按钮               |
| timeout | 超时               | 超时提示、重试按钮、可取消说明   |

推荐封装 `useAsyncState` 或 `useRequestState`，减少页面重复逻辑。

## 9. 枚举与展示标签

所有枚举值从契约第 3 节抽离到 `src/constants/enums.js`：

```js
export const MealType = ['BREAKFAST', 'LUNCH', 'DINNER', 'SNACK']
export const Difficulty = ['EASY', 'NORMAL', 'HARD']
export const RiskSeverity = ['INFO', 'WARNING', 'DANGER']
```

中文展示放在 `src/constants/labels.js`：

```js
export const MealTypeLabel = {
  BREAKFAST: '早餐',
  LUNCH: '午餐',
  DINNER: '晚餐',
  SNACK: '加餐',
}
```

原则：

- 请求和响应永远使用契约枚举值。
- 页面展示通过 Label 映射转换。
- 表单选项由枚举和 Label 生成，不手写多份下拉选项。

## 10. 联调流程

推荐联调顺序：

1. 先联调认证：`/auth/login`、`/auth/me`、`40100`。
2. 再联调只读接口：健康档案查询、菜谱列表、营养今日汇总。
3. 再联调写接口：档案保存、收藏、反馈、评论。
4. 最后联调耗时接口：AI 解析、计划生成、报告生成、风险评估。

逐接口切换流程：

1. 在 `endpointDataSource` 将目标接口从 `MOCK` 改为 `REAL`。
2. 保留同模块其他接口为 `MOCK`，确保页面仍可独立开发。
3. 使用契约示例和 Mock 数据对比真实响应字段。
4. 若真实字段偏离契约，优先修后端或更新契约，不在 UI 层写临时兼容。
5. 接口稳定后补充一条回归记录到模块联调清单。

## 11. 质量验收清单

基础设施：

- `request` 自动注入 token。
- `code !== 0` 会进入统一错误流程。
- `40100` 会清空登录态并跳转登录页。
- `VITE_USE_MOCK=false` 时不注册 Mock.js。
- 单接口可独立切换 Mock/真实 API。

Mock：

- 每个接口均支持正常、空、错误、超时场景。
- Mock 数据字段名、枚举值、分页结构与契约一致。
- Mock 工厂不在页面组件中引用。
- 列表、详情、图表、表单回填均使用随机数据验证过。

页面：

- 所有列表页支持 loading、empty、error、pagination。
- 所有写操作有提交中状态和失败提示。
- AI、报告、风险执行等耗时操作有超时和重试。
- 管理员页面有角色守卫。
- 游客可访问社区列表和详情，受保护操作会引导登录。

联调：

- 已联调接口在 `endpointDataSource` 中切到 `REAL`。
- 未联调接口保持 `MOCK`，不阻塞页面开发。
- 真实响应与契约不一致时记录差异并回写契约或后端任务。
